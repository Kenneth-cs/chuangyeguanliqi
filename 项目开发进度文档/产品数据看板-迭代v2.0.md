# 产品数据看板 — 迭代 v2.0

> **迭代背景：** 随着接入项目增多（钱小满、人生教练等），原「功能分析」Tab 为钱小满硬编码，每新增一个项目都需要手动开发定制看板，维护成本过高。
>
> **本次迭代目标：** 用「通用参数分析」替代硬编码「功能分析」，实现零配置接入；同时新增 Excel 数据导出功能，方便离线深度分析。

## 二、功能一：通用「参数分析」Tab

### 2.1 背景与决策

**现状问题：**
- 当前「功能分析」Tab 写死了钱小满的两张表（项目类型分布、金额区间分布）
- 每个新项目要新增定制看板，预计 2-4 小时/项目，随项目增多不可持续

**为什么不需要定制化：**
- `eventName` 上报时已携带中文名（如「查看卦象结果」），系统直接读取，天然可读
- `params` 的 value 很多本身是中文字符串（如 `scenario: "职场/感情/投资"`），分布统计无需翻译
- 只有少数 key 是英文（如 `plan_type`、`wait_time_ms`），英文 key 对数据分析者已足够

**「事件明细」Tab 与「参数分析」Tab 的分工：**

| | 事件明细 | 参数分析 |
|---|---|---|
| 数据粒度 | 原始流水，每行一条记录 | 聚合统计，分布视角 |
| 适合回答 | 某设备今天触发了哪些事件？ | 用户最常选哪个场景？ |
| 类比 | 数据库原始表 | BI 统计报告 |

### 2.2 功能设计

**Tab 位置：** 替换原「功能分析」Tab，Tab 名称改为「参数分析」

**交互流程：**

```
用户进入「参数分析」Tab
  ↓
顶部下拉菜单：选择事件（从数据库自动扫描该项目上报过的所有事件）
  - 选项格式：「记账成功（record_submit_success）」
  - 默认选中上报次数最多的事件
  ↓
自动展示该事件所有 params key 的值分布
  - 字符串类 value → 条形图 + 计数表
  - 数字类 value → 均值 / 最大值 / 最小值 + 区间分布表
  ↓
切换事件时，分布结果实时更新
```

### 2.3 后端接口设计

**新增接口：**
```
GET /api/analytics/params?projectId=xxx&range=7d&eventId=view_hexagram_result
```

**返回结构：**
```json
{
  "eventId": "view_hexagram_result",
  "eventName": "查看卦象结果",
  "totalCount": 2948,
  "params": [
    {
      "key": "scenario",
      "type": "string",
      "distribution": [
        { "value": "职场", "count": 1240, "ratio": 0.42 },
        { "value": "感情", "count": 914,  "ratio": 0.31 },
        { "value": "投资", "count": 589,  "ratio": 0.20 },
        { "value": "自定义", "count": 205, "ratio": 0.07 }
      ]
    },
    {
      "key": "wait_time_ms",
      "type": "number",
      "stats": { "avg": 2340, "max": 8120, "min": 450 },
      "distribution": [
        { "label": "< 1000ms",    "count": 442,  "ratio": 0.15 },
        { "label": "1000-3000ms", "count": 1710, "ratio": 0.58 },
        { "label": "3000-5000ms", "count": 590,  "ratio": 0.20 },
        { "label": "> 5000ms",    "count": 206,  "ratio": 0.07 }
      ]
    }
  ]
}
```

**获取事件列表接口（供下拉菜单）：**
```
GET /api/analytics/event-list?projectId=xxx&range=7d
```

返回：
```json
[
  { "eventId": "record_submit_success", "eventName": "记账成功", "count": 4000 },
  { "eventId": "record_click_add",      "eventName": "点击记一笔入口", "count": 5500 }
]
```

**类型判断逻辑（后端）：**
- 取最近 20 条该 key 的 value 样本
- 如果全部能 `parseFloat` → 识别为 `number` 类型，走均值+区间分布
- 否则识别为 `string` 类型，走值分布统计
- `number` 类型的区间分界点自动计算（四分位数划分）

## 三、功能二：Excel 数据导出

### 3.1 功能设计

**入口位置：** 每个项目看板右上角，全局筛选器旁边

```
[核心大盘] [留存矩阵] [转化路径] ...        [↓ 导出数据]
时间范围: [近7天 ▾]   版本: [全部 ▾]
```

**导出弹窗交互：**

```
┌─────────────────────────────────────────┐
│  导出数据                            ✕  │
│                                         │
│  导出方式                               │
│  ○ 按时间范围导出（当前筛选器设置）      │
│  ● 选择具体日期导出                     │
│                                         │
│  选择日期（可多选）                      │
│  [日历组件 - 可点选单个或多个日期]       │
│  已选：2026-04-18, 2026-04-19, 2026-04-20│
│                                         │
│  导出内容                               │
│  ☑ 原始事件流水（事件明细）             │
│  ☑ 每日汇总（核心大盘数据）             │
│  ☐ 留存矩阵                            │
│                                         │
│  [取消]              [导出 Excel ↓]     │
└─────────────────────────────────────────┘
```

**导出的 Excel 结构（多 Sheet）：**

| Sheet 名 | 内容 |
|---------|------|
| 事件流水 | 原始 AppEvent 数据（日期/设备ID/事件名/事件ID/params展开列/版本） |
| 每日汇总 | 日期/DAU/新增用户/事件总次数/激活率/核心渗透率 |
| 留存矩阵 | 安装日期/新增设备/D1/D3/D7/D30 |

**事件流水 Sheet 的 params 处理：**
- `params` 字段自动拆分为多列（每个 key 一列）
- 例：`params: {"category": "餐饮", "amount_level": "level_1"}` → 拆为 `params.category` 列和 `params.amount_level` 列
- 不同事件的 params 不同，缺失字段留空

### 3.2 后端接口设计

**新增接口：**
```
POST /api/analytics/export
Content-Type: application/json
```

请求体：
```json
{
  "projectId": "xxx",
  "mode": "dates",
  "dates": ["2026-04-18", "2026-04-19", "2026-04-20"],
  "sheets": ["events", "daily", "retention"]
}
```

或时间范围模式：
```json
{
  "projectId": "xxx",
  "mode": "range",
  "range": "7d",
  "sheets": ["events", "daily"]
}
```

**返回：** 直接返回 `.xlsx` 文件流（`Content-Disposition: attachment; filename="项目名_20260420.xlsx"`）

**依赖库：** `xlsx`（SheetJS，纯 Node.js，无系统依赖）

安装：
```bash
npm install xlsx
```

## 四、Tab 结构最终版

```
核心大盘 / 留存矩阵 / 转化路径 / 参数分析 / 事件明细 / API 接入 / 分享管理
```

| Tab | 类型 | 是否需要按项目定制 | 说明 |
|-----|------|--------------|------|
| 核心大盘 | 固定 | 否 | 通用指标（DAU/新增/留存） |
| 留存矩阵 | 固定 | 否 | 通用算法 |
| 转化路径 | 半通用 | 可选 | 默认展示最常见事件链路，可手动指定步骤 |
| **参数分析** | **动态自动** | **否** | 自动扫描 params，零配置 |
| 事件明细 | 固定 | 否 | 原始流水，天然通用 |
| API 接入 | 固定 | 否 | 通用 |
| 分享管理 | 固定 | 否 | 通用 |

**结论：新项目接入后，无需任何额外开发，所有 Tab 自动可用。**

## 六、后续可选扩展（v3.0 备用）

- **参数分析 — 按日趋势**：选定某个 params key 的某个 value，看它随时间的变化趋势（如「选择职场的用户近 30 天的每日数量趋势」）
